## ADDED Requirements

### Requirement: All HTTP error responses are JSON with an error code

The system SHALL return every HTTP error response (4xx and 5xx, framework-level or handler-level) as `application/json` with a body that begins with an `error` field whose value is a stable string code. Additional fields MAY accompany `error` (e.g. `hint`, `barcode`, `product_id`); they MUST NOT replace it.

#### Scenario: Known handler errors use the JSON shape

- **WHEN** a handler validates input and rejects it (e.g. `POST /meals` with missing `product_id`)
- **THEN** the response status is the appropriate 4xx
- **AND** the response `Content-Type` begins with `application/json`
- **AND** the response body parses as a JSON object containing an `error` field whose value is a known code (e.g. `product_id_required`)

#### Scenario: Unknown routes return JSON

- **WHEN** the client sends a request whose method+path does not match any registered route (e.g. `GET /this/does/not/exist`)
- **THEN** the response status is `404 Not Found`
- **AND** the response `Content-Type` begins with `application/json`
- **AND** the response body is `{"error":"not_found"}`

#### Scenario: Known routes with the wrong method return JSON

- **WHEN** the client sends a method that does not match any handler registered at an otherwise-known path (e.g. `PATCH /healthz`)
- **THEN** the response status is `405 Method Not Allowed`
- **AND** the response `Content-Type` begins with `application/json`
- **AND** the response body is `{"error":"method_not_allowed"}`

#### Scenario: Framework-level errors never leak plain text

- **WHEN** any HTTP error response is produced — including those generated by the routing framework's default handlers — for any path under the API
- **THEN** the response body is valid JSON
- **AND** is NOT the framework's default plain-text body (e.g. `"404 page not found"`)
